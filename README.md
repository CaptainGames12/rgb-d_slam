**1. Introduction**

Simultaneous Localization and Mapping (SLAM) is one of the fundamental
problems in robotics and computer vision. The goal of a SLAM system is
to estimate the position of a moving sensor while simultaneously
building a map of the surrounding environment. SLAM algorithms are
widely used in autonomous vehicles, mobile robots, augmented reality
systems, and 3D reconstruction applications.

In this project, different RGB-D SLAM methods were implemented and
evaluated using the ScanNet dataset. Instead of collecting data using a
physical camera, pre-recorded RGB-D sequences from ScanNet were used.
These sequences contain synchronized colour images, depth images, camera
calibration information, and ground-truth camera poses.

The objective of the project was to integrate SLAM algorithms into ROS
2, process the RGB-D data stream, estimate the camera trajectory, and
compare the estimated trajectory against the ground-truth trajectory
provided by ScanNet.

Each group member implemented and evaluated a different SLAM method. The
estimated trajectories generated by the individual methods were compared
against the ground-truth trajectories provided by the ScanNet dataset.

**2. Problem Statement**

The main task was to estimate the trajectory of a moving RGB-D camera
using only the information provided by the ScanNet dataset.

The estimated camera poses generated by the SLAM algorithm were compared
against the true camera poses available in the dataset. The quality of
the trajectory estimation was evaluated using Absolute Trajectory Error
(ATE), Root Mean Square Error (RMSE), maximum trajectory error, and
standard deviation.

The project also involved investigating how different algorithm
parameters influence both trajectory accuracy and computational
performance.

The main objectives of this project were:

**3. Background**

**3.1 RGB -- D Cameras**

RGB-D cameras provide both colour information (RGB) and depth
information for every pixel in an image.

The RGB image contains appearance information, while the depth image
contains the distance between the camera and the observed scene.
Together, these measurements allow a three-dimensional representation of
the environment to be reconstructed.

Using the camera intrinsic parameters, each depth pixel can be
transformed into a 3D point, producing a point cloud representation of
the scene.

An RGB-D camera provides:

-   RGB image (colour information)

-   Depth image (distance information)

The depth image allows every pixel to be converted into a
three-dimensional point.

Using camera intrinsics:

-   focal length fx

-   focal length fy

-   principal point cx

-   principal point cy

A depth pixel can be projected into 3D space.

Although different SLAM methods use different approaches, the general
pipeline typically consists of:

1.  Receiving RGB and depth images.

2.  Extracting useful information from the images.

3.  Estimating motion between consecutive frames.

4.  Updating the camera trajectory.

5.  Building or maintaining a map of the environment.

The accuracy of the final trajectory depends on factors such as sensor
quality, feature extraction, point cloud registration, optimization
techniques, and parameter selection.

**Method Overview**

Open3D-SLAM is a library built specifically for trajectory estimation
using 3D engine Open3D for working with 3-dimensional coordinates and
3D-graphics. The method used inside the library for map building is
generalized ICP. The library is deprecated and built for old ROS so I
have adapted it to ROS2 and implemented lua config files.

**Generalized Iterative Closest Point (G-ICP)** is an advanced algorithm
used to align 3D point clouds. While standard ICP often struggles with
noisy data or featureless environments, G-ICP provides a more robust
mathematical framework by treating the alignment as a probabilistic
surface-to-surface registration problem.

### **The Core Concept**

When aligning raw LiDAR scans---whether for real-time SLAM pipelines or
post-processing in spatial data libraries like Open3D---standard
algorithms usually pair points based on simple geometry. G-ICP upgrades
this logic:

-   **Generalized ICP:** Unifies these approaches by modeling the local
    > neighborhood of *every* point (in both the source and target
    > clouds) as a Gaussian distribution.

**4.2 Method 2**

#### **1. Objective**

The goal of the project is to integrate point-cloud registration
libraries as ROS 2 nodes for camera pose tracking (visual odometry /
SLAM) and to evaluate their accuracy against ground-truth camera
trajectories. Instead of a live camera, the input is taken from
pre-recorded logs of the **ScanNet** dataset, which provide synchronised
colour images, depth images, camera intrinsics, and ground-truth camera
poses.

This particular method uses **small_gicp** --- a highly optimised,
header-only C++ library (with Python bindings) implementing fine
point-cloud registration algorithms (ICP, Point-to-Plane ICP, GICP,
VGICP). The deliverable is a ROS 2 node that:

1.  subscribes to the depth image stream and camera intrinsics published
    > from a ScanNet log,

2.  reconstructs a 3D point cloud from each depth frame,

3.  estimates the incremental camera motion by registering successive
    > observations,

4.  accumulates these motions into a full camera trajectory, and

5.  exports that trajectory in TUM format so it can be quantitatively
    > compared to ground truth.

The trajectory is evaluated across multiple ScanNet scenes using the
**Absolute Trajectory Error (ATE)** and **Relative Pose Error (RPE)**
metrics.

## **4. ROS 2 System Architecture**

The pipeline runs as two cooperating ROS 2 processes.

### **4.1 The Publisher (scannet_publisher)**

The publisher reads a .sens log and republishes its contents as standard
ROS 2 topics, emulating a live sensor:

-   /camera/depth/image_raw --- depth images (sensor_msgs/Image)

-   /camera/depth/camera_info --- depth camera intrinsics
    > (sensor_msgs/CameraInfo)

-   /camera/color/image_raw (+ compressed variants) --- colour images

-   /camera_pose --- ground-truth camera pose
    > (geometry_msgs/PoseStamped)

-   /clock --- simulation clock

Internally it reads frames in a background thread and publishes them at
a fixed rate (30 Hz by default).

### **4.2 The SLAM Node (slam_node.py)**

The SLAM node is the core contribution. It subscribes to:

-   /camera/depth/camera_info --- to obtain the pinhole intrinsics fx,
    > fy, cx, cy,

-   /camera/depth/image_raw --- the depth stream to be processed,

-   /camera_pose --- the ground-truth pose stream, recorded **only** for
    > later evaluation and **never** used to inform the estimate.

For each depth frame the node reconstructs a point cloud, registers it
to estimate camera motion, composes that motion into the running pose,
and stores both the estimate and the corresponding ground-truth pose. On
shutdown it writes two trajectory files in TUM format.

## **5. SLAM Methodology**

### **5.1 Depth-to-Point-Cloud Conversion**

Each incoming depth image is converted from raw 16-bit millimetre values
to metres and filtered to a valid sensing range:

  -----------------------------------------------------------------------
  depth = depth.astype(np.float32) / 1000.0 \# millimetres → metres\
  depth\[depth \< 0.1\] = 0 \# remove too-near / invalid returns\
  depth\[depth \> 10.0\] = 0 \# remove too-far / unreliable returns
  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

The valid depth pixels are then back-projected into 3D using the
standard pinhole camera model. For a pixel (u, v) with depth z:

  -----------------------------------------------------------------------
  x = (u − cx) · z / fx\
  y = (v − cy) · z / fy\
  z = z
  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

In vectorised form:

  -----------------------------------------------------------------------
  def depth_to_pointcloud(self, depth):\
  h, w = depth.shape\
  u, v = np.meshgrid(np.arange(w), np.arange(h))\
  x = (u - self.cx) \* depth / self.fx\
  y = (v - self.cy) \* depth / self.fy\
  mask = depth \> 0\
  points = np.stack(\[x\[mask\], y\[mask\], depth\[mask\]\], axis=-1)\
  return points.astype(np.float32)
  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

The boolean mask discards the zeroed-out invalid pixels, yielding an (N,
3) array of valid 3D points in the camera frame. Frames yielding fewer
than 100 points are skipped as degenerate.

### **5.2 Point Cloud Preprocessing**

Raw per-frame clouds contain hundreds of thousands of points, which is
both noisy and computationally prohibitive for registration. Each cloud
is therefore preprocessed in a single parallelised small_gicp call:

  -----------------------------------------------------------------------
  def depth_to_pointcloud(self, depth):\
  h, w = depth.shape\
  u, v = np.meshgrid(np.arange(w), np.arange(h))\
  x = (u - self.cx) \* depth / self.fx\
  y = (v - self.cy) \* depth / self.fy\
  mask = depth \> 0\
  points = np.stack(\[x\[mask\], y\[mask\], depth\[mask\]\], axis=-1)\
  return points.astype(np.float32)
  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

This performs three operations:

1.  **Voxel downsampling** at DOWNSAMPLE_RESOLUTION --- points are
    > bucketed into a voxel grid and replaced by representative points,
    > drastically reducing cloud size while preserving geometry.

2.  **KdTree construction** --- a spatial index over the downsampled
    > cloud for fast nearest-neighbour queries.

3.  **Local covariance estimation** --- for each point, the covariance
    > of its NUM_NEIGHBORS nearest neighbours is computed. These
    > covariances describe the local surface distribution and are what
    > make the *Generalized* ICP (GICP) registration possible.

### **5.3 Registration (Generalized ICP)**

Camera motion is recovered by registering the current observation
against a reference using GICP. GICP generalises classical ICP by
modelling each point as a local Gaussian distribution (via the
covariances above) rather than a bare 3D point, and minimising a
distribution-to-distribution distance. This makes it markedly more
robust on noisy depth data and on the partially overlapping, featureless
regions common in indoor scenes than point-to-point ICP.

The registration call returns a RegistrationResult containing the
estimated rigid transform T_target_source:

  -----------------------------------------------------------------------
  result = small_gicp.align(\
  target, source,\
  init_T_target_source = \<initial guess\>,\
  max_correspondence_distance = MAX_CORR_DIST,\
  num_threads = NUM_THREADS,\
  max_iterations = MAX_ITERATIONS,\
  )
  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

MAX_CORR_DIST rejects correspondences further apart than the threshold,
controlling robustness to outliers and to large inter-frame motion.
MAX_ITERATIONS caps the iterative optimisation.

### **5.4 Pose Composition**

The camera pose is maintained as a 4×4 homogeneous transform
current_pose, initialised to the identity (the origin of the
trajectory). **How** the registration result is composed into this pose
depends on whether the reference is the previous frame (scan-to-scan) or
the accumulated map (scan-to-map). This distinction is the central
methodological theme of the project and is detailed in Section 7.

For each successfully processed frame, the timestamp (from the depth
message header) and a copy of the current pose are appended to parallel
lists, which are later written to disk.

## **6. Real-Time Decoupling: The Frame-Dropping Problem**

A significant engineering problem arose from the mismatch between data
rate and compute throughput.

**The problem.** The publisher emits depth frames at 30 Hz. On the ARM64
virtual machine, each frame requires point-cloud reconstruction,
downsampling, KdTree construction, covariance estimation, and GICP
optimisation --- taking substantially longer than the 33 ms budget per
frame. ROS 2 subscriptions buffer incoming messages in a fixed-size
queue; once that queue is full, the **oldest unprocessed messages are
silently discarded**. With the default small queue, the SLAM node fell
behind almost immediately and processed only a fraction of the sequence
(e.g. \~1300 of 5578 frames), producing an incomplete and badly drifted
trajectory.

Simply enlarging the subscription queue postponed but did not solve the
problem: because the *rate* mismatch is fundamental, any finite queue
eventually overflows over a long sequence.

**The solution --- producer/consumer decoupling.** The architecture was
restructured so that *receiving* and *processing* are fully decoupled:

-   The depth subscription callback does nothing but place the incoming
    > message onto an **unbounded, thread-safe Python queue**
    > (queue.Queue()). This callback returns essentially instantly, so
    > the node never falls behind the publisher and no message is ever
    > dropped.

-   A dedicated **worker thread** consumes messages from this queue one
    > at a time, performing the full SLAM computation at whatever pace
    > the hardware allows.

  -----------------------------------------------------------------------
  def depth_cb(self, msg):\
  self.depth_queue.put(msg) \# enqueue instantly, never blocks, never
  drops\
  \
  def process_loop(self): \# runs in a separate worker thread\
  while rclpy.ok():\
  msg = self.depth_queue.get(timeout=1.0)\
  self.process_frame(msg)\
  self.depth_queue.task_done()
  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

On shutdown, self.depth_queue.join() is called before writing results,
forcing the node to finish processing every buffered frame even after
the publisher has stopped.

The net effect is that the publisher transfers the entire sequence into
memory quickly, and the worker thread then drains the backlog to
completion. This guarantees **all** frames are processed regardless of
per-frame cost, at the price of total runtime exceeding real time. After
this change the node consistently processes essentially the full
sequence (e.g. \~5346 of 5578 frames for scene0000_00).

> *Implementation note:* on Ctrl+C, rclpy emits a benign rcl_shutdown
> already called traceback as the context is torn down twice. It does
> not affect the saved results and is ignored. A NumPy version warning
> printed at startup is likewise harmless.

## **7. From Scan-to-Scan to Scan-to-Map (Core Methodological Advance)**

The single most important design decision in this project is the
**registration reference**, which directly determines trajectory drift.

### **7.1 Scan-to-Scan Registration and Its Drift**

The initial implementation used **scan-to-scan** registration: each new
frame was registered against only the *immediately preceding* frame. The
global pose was then updated by right-multiplying the incremental
motion:

$current\_ pose\  \leftarrow \ current\_ pose\  \cdot \ T\_ target\_ source$

This is the simplest possible odometry formulation, but also the most
drift-prone. Every pairwise registration carries a small error; because
each pose is built by chaining these increments, the errors **accumulate
monotonically** over the sequence, and --- critically --- there is no
mechanism for them ever to be corrected, because past geometry is
discarded after each step.

The empirical signature was clear in the diagnostic plots: ATE stayed
near zero for the first portion of a sequence and then grew steadily,
with a pronounced **scale / Z-axis drift** in scenes where the camera
moved little vertically (little geometry to constrain depth). The error
distribution became distinctly **bimodal** --- a cluster of well-tracked
early frames and a cluster of drifted later frames.

### **7.2 Scan-to-Map Registration with a Voxel Map**

The decisive improvement was switching to **scan-to-map** registration.
Instead of registering against a single previous frame, each new
observation is registered against an **incrementally built map of all
geometry observed so far**, and is then merged into that map.

small_gicp provides a GaussianVoxelMap for exactly this purpose: a
voxelised structure in which each occupied voxel stores a Gaussian (mean
and covariance) summarising the points that have fallen into it.
Registering against this map means each frame is corrected against the
*aggregate* of all prior observations rather than just the last one,
which strongly suppresses drift --- the map effectively \"remembers\"
the correct scene geometry.

The per-frame logic becomes:

  -----------------------------------------------------------------------
  \# initialise the map on the first valid frame\
  if self.voxelmap is None:\
  self.voxelmap = small_gicp.GaussianVoxelMap(VOXEL_LEAF_SIZE)\
  self.voxelmap.insert(pcd, self.current_pose)\
  return\
  \
  \# subsequent frames: register against the map, then update the map\
  result = small_gicp.align(\
  self.voxelmap, pcd,\
  init_T_target_source = self.current_pose,\
  max_correspondence_distance = MAX_CORR_DIST,\
  num_threads = NUM_THREADS,\
  max_iterations = MAX_ITERATIONS,\
  )\
  self.current_pose = result.T_target_source \# absolute pose in the map
  frame\
  self.voxelmap.insert(pcd, self.current_pose) \# fold the new frame into
  the map
  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

Two points distinguish this from the scan-to-scan formulation:

-   The previous pose is supplied as the **initial guess**
    > (init_T_target_source = self.current_pose), giving the optimiser a
    > warm start near the true solution.

-   The result T_target_source is the **absolute** camera pose in the
    > map frame, so current_pose is assigned directly rather than
    > composed multiplicatively. The newly registered cloud is then
    > inserted into the map (transformed by that absolute pose) so the
    > map grows with the trajectory.

This change reduced first-pose-aligned mean ATE from \~0.84 m
(scan-to-scan) to \~0.40 m (scan-to-map) on scene0000_00, and produced
consistently strong results across scenes (Section 10). The diagnostic
plots confirm the mechanism: error still rises during difficult passages
(fast rotation, feature-poor regions) but, unlike scan-to-scan,
**partially recovers** afterwards because the map re-anchors the
estimate.

### **7.3 GaussianVoxelMap versus IncrementalVoxelMap**

small_gicp exposes both a GaussianVoxelMap and an IncrementalVoxelMap
(the latter additionally supporting LRU-based voxel eviction for
unbounded operation). During development the node was first written
against IncrementalVoxelMap, but small_gicp.align only accepts a
GaussianVoxelMap as a registration target --- its overload set covers
PointCloud + KdTree targets and a GaussianVoxelMap target, but not
IncrementalVoxelMap. The node was therefore implemented with
GaussianVoxelMap, which is the correct map type for the registration
interface and is sufficient for the bounded-length ScanNet sequences
used here.

## **8. Parameter Tuning**

The registration parameters were tuned empirically on scene0000_00,
using evo (Sim3-aligned APE) as the objective. The progression was:

  ----------------------------------------------------------------------------------
  **Round**   **Downsample   **Neighbours**   **Max corr.     **Mean APE   **RMSE
              (m)**                           dist. (m)**     (m)**        (m)**
  ----------- -------------- ---------------- --------------- ------------ ---------
  Baseline    default        default          default         0.730        0.849

  Round 1     0.02           20               0.5             0.808        0.877

  Round 2     0.05           30               0.3             0.552        0.610

  Round 3     0.05           50               0.2             0.531        0.589
  ----------------------------------------------------------------------------------

The clear trend is that a **coarser downsampling resolution (0.05 m)**,
**more neighbours (50)** for covariance estimation, and a **tighter
maximum correspondence distance (0.2 m)** all improve accuracy. Coarser
downsampling yields more stable, less noise-dominated surfaces; more
neighbours give better-conditioned local covariances; and a tighter
correspondence gate rejects spurious matches. Round 1 performed worse
than baseline because 0.02 m resolution was too dense for the ARM64 VM,
causing the node to fall behind and drop frames before the
producer/consumer redesign was in place.

The final parameter set, carried over to the scan-to-map evaluation
across all scenes, is:

  -----------------------------------------------------------------------
  DOWNSAMPLE_RESOLUTION = 0.05 \# voxel size for per-frame downsampling
  (m)\
  NUM_NEIGHBORS = 50 \# neighbours for covariance estimation\
  MAX_CORR_DIST = 0.20 \# correspondence rejection threshold (m)\
  NUM_THREADS = 4 \# parallel worker threads\
  MAX_ITERATIONS = 50 \# GICP iteration cap\
  VOXEL_LEAF_SIZE = 0.05 \# voxel size of the GaussianVoxelMap (m)
  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

## **9. Evaluation Methodology**

### **9.1 TUM Trajectory Format**

Both the estimated and ground-truth trajectories are written in the
**TUM format**, one pose per line:

  -----------------------------------------------------------------------
  timestamp tx ty tz qx qy qz qw
  -----------------------------------------------------------------------

  -----------------------------------------------------------------------

i.e. a timestamp, a translation vector, and an orientation quaternion.
The node converts each stored 4×4 pose to this representation (rotation
matrix → quaternion via SciPy) and writes \~/estimated_tum.txt and
\~/groundtruth_tum.txt. Because the estimate and ground truth are
recorded from the same frame stream, the two files contain a matched
pose for (almost) every processed frame.

### **9.2 Absolute Trajectory Error (ATE)**

ATE measures the global consistency of the estimated trajectory. The
estimated trajectory is first aligned to ground truth (Section 9.4);
then for each ground-truth pose the temporally closest estimated pose
(within a 100 ms tolerance) is found, and the error is the Euclidean
distance between their translation components:

$e_{i}\  = \ \|\ t_{gt,\ i}\  - \ t_{est,\ i}\|$

The evaluator reports the **mean, median, standard deviation, and
maximum** of these per-frame errors. ATE is the headline metric because
it directly quantifies how far, on average, the estimated camera
position deviates from the true position over the whole sequence.

### **9.3 Relative Pose Error (RPE)**

RPE measures *local* consistency --- the accuracy of incremental motion
rather than the global trajectory. For consecutive ground-truth pose
pairs separated by more than 0.1 m, the relative transform ΔT_gt = T₁⁻¹
T₂ is compared to the corresponding estimated relative transform,
yielding a translational error (per metre travelled) and a rotational
error (degrees per metre). In the current evaluation the RPE values are
reported as nan; this stems from strict timestamp matching between the
estimated and ground-truth streams and does not affect the ATE results,
which are the primary metric of interest.

### **9.4 Trajectory Alignment: First-Pose versus Sim3 Umeyama**

The reported error depends heavily on **how** the estimated trajectory
is aligned to ground truth before comparison. Two conventions were used,
answering different questions:

**(a) First-pose alignment** (the group\'s custom
evaluate_trajectory.py). The estimate is locked to ground truth at the
very first pose only:

$T_{align\ } = \ T_{gt,\ \Theta}*T_{est,\ \Theta^{- 1}}$

This single transform is applied to all estimated poses, with no scale
correction. It is a strict metric: any accumulated rotational or scale
drift is fully exposed, because nothing downstream of the first pose is
re-fitted.

**(b) Sim3 Umeyama alignment** (evo_ape \... \--align \--correct_scale).
The estimate is aligned by the best-fit similarity transform over the
*entire* trajectory --- optimal rotation, translation, **and a global
scale factor**. This is more forgiving: a slow, consistent scale drift
is largely absorbed by the fitted scale, so the reported error is lower.

The gap between the two metrics is itself diagnostic. For the
scan-to-scan trajectory, first-pose alignment gave \~0.84 m while Sim3
gave \~0.63 m on the same data --- the discrepancy revealing that
accumulated scale / global drift, rather than local jitter, was the
dominant error. This insight directly motivated the move to scan-to-map
registration. **The primary metric used to assess the method is the
stricter first-pose-aligned ATE.**

### **9.5 Visualisation**

For each scene, six diagnostic plots are generated from the two TUM
files:

1.  **3D trajectory comparison** --- estimated vs ground-truth path in

2.  **2D projections** --- top-down (XY) and side (XZ, YZ) views, plus
    > distance-travelled vs frame.

3.  **Per-axis position components** --- X, Y, Z over time, isolating
    > directional drift (the Z component exposes depth / scale drift).

4.  **Orientation components** --- roll, pitch, yaw over time.

## **11. Discussion and Analysis**

The per-scene diagnostic plots make the method\'s behaviour concrete.

On **scene0000_00**, XY tracking is tight throughout and the **scale is
correct** --- the distance-travelled curves of estimate and ground truth
nearly overlap, and the yaw estimate tracks ground truth through all
±180° wraps. The residual error is dominated by a small, consistent **Z
(depth) bias** developing in the latter half of the sequence, where the
camera\'s limited vertical motion leaves the depth direction weakly
constrained. This appears as a bimodal ATE histogram (a well-tracked
cluster near 0.1--0.2 m and a drifted cluster near 0.5--0.6 m).

On **scene0011_00**, the error stays below 0.2 m for roughly the first
40 s, spikes to \~1.0 m during a difficult passage around 50 s (likely
fast rotation or a feature-poor region), and then **partially recovers**
to \~0.4 m. The median of 0.17 m confirms that most of the trajectory is
tracked to within \~17 cm; the mean is pulled up only by the transient
spike. The recovery after the spike is the defining advantage of
scan-to-map over scan-to-scan: scan-to-scan would have retained that
error permanently, whereas the voxel map re-anchors the estimate.

The two weaker scenes --- **scene0011_01** (1.5410 m) and
**scene0050_01** (0.6651 m) --- illustrate the limits of a pure odometry
pipeline. In sequences with rapid camera motion, repetitive geometry, or
extended low-overlap viewing, registration errors can accumulate faster
than the voxel map can correct for them. Without loop closure or
pose-graph optimisation there is no global mechanism to recover drift
accumulated during such passages once the camera revisits a previously
mapped region. These scenes represent the natural failure mode of
frame-to-map registration in the absence of a global optimisation
back-end, and are the principal motivation for the future work in
Section 14.

**4.3 Method 3\
PCL ICP RGB-D SLAM**

**By - Swayam Virmani**

**Method Overview**

This method implements a complete RGB-D Simultaneous Localization and
Mapping (SLAM) pipeline using the Iterative Closest Point (ICP)
algorithm provided by the Point Cloud Library (PCL). The objective of
this implementation is to estimate the camera trajectory by registering
consecutive RGB-D frames captured from the ScanNet dataset.

Each incoming depth image is converted into a 3D point cloud using the
camera intrinsic parameters. Before registration, the point cloud is
downsampled using a voxel grid filter to reduce computational complexity
while preserving the overall geometry. The ICP algorithm is then applied
to estimate the rigid transformation between consecutive point clouds.
These frame-to-frame transformations are accumulated to reconstruct the
complete camera trajectory throughout the sequence.

Unlike feature-based SLAM methods, this implementation relies entirely
on geometric alignment of point clouds. Consequently, the approach is
simple and computationally efficient, but its performance depends
heavily on the quality of point cloud overlap and does not include loop
closure or global optimization, leading to drift over long trajectories.

**About ICP?**

The Iterative Closest Point (ICP) algorithm is one of the most widely
used methods for rigid point cloud registration. Its objective is to
estimate the transformation that best aligns two overlapping point
clouds by minimizing the distance between corresponding points.

For each pair of consecutive RGB-D frames, ICP performs the following
steps:

1.  Find the closest corresponding points between the source and target
    > point clouds.

2.  Estimate the rigid transformation (rotation and translation) that
    > minimizes the point-to-point registration error.

3.  Apply the estimated transformation to the source point cloud.

4.  Repeat the process until convergence or until the maximum number of
    > iterations is reached.

The estimated transformation is then accumulated with the previous
camera pose to construct the complete trajectory.
